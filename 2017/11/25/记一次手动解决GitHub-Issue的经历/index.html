记一次手动解决GitHub Issue的经历

Nov 25, 2017 | Hits

前题

大概两周前，我们伟大的老板振臂一呼，指派我一个任务: Android设备触摸事件转OTG输出。说通俗一点，就是把你的Android设备变成一个鼠标，通过编程再实现操控PC的效果。作为一个Android开发工程师，我的内心是崩溃的，颤抖的打开老板发我的链接android-keyboard-gadget。看了一遍REAMDE,内心OS:先把Demo跑通吧。就这样硬着头皮上了。

前期准备

了解原理

Android系统建立在Linux内核上，而Linux通过usb-gadget提供了把设备模拟成键盘、和鼠标的支持。
什么是usb-gadget ？当把pad/手机插到pc上时，可以作为u盘、网卡等usb功能设备呈现，这个就叫做gadget。
有了这个前提，修改Kernel，我们可以把Android设备编程一个鼠标，这个小目标就可以实现了。

DEMO的运行

1. 寻找支持设备

这个是很重要的东西。
在刚开始接触的时候，感觉很模糊，还不太清楚自己面对的这个东西，也不能确定，这个Demo是不是我们想要的。这个时候，应该最快的速度先把Demo跑通，试一下效果吗！
这个勤劳的github主和他的小伙伴提供了大概27个机型编译后的Kernel、Boot或者ROM. 我翻了一遍，都是一些比较old的设备，整个公司(我们也就20个工程师) 也只能找出一台三星 S4. 遗憾的是，经历了一个下午的折腾，并没有成功的刷入新的Boot.img. 我忽然想起我的老东家，当时还在为它搬砖的时候，申请过一台nexus的测试机。于是联系了一下原来的同事，刚好是我需要的nexus 5 。Nice!

2. 刷机

步骤如下：

  1. 进入fastboot模式
  2. 刷入准备好的img文件
  3. 重启
    可能用到的命令行给大家列举一下：
    adb reboot fastboot : 让设备重启并进入fastboot模式
    fastboot devices: 列出已连接的fastboot模式的设备
    fastboot flash boot boot.img : 刷入boot文件
    fastboot flash kernel kernel.img : 刷入kernel文件
    嗯，这个过程我又学习到一些刷机的知识，还有一些新发现。比如这个fastboot的命令，我居然可以直接使用。这是为什么呢？会不会和adb有关系？打开xxx/sdk/platform-tools/这个目录，我就看到一个fastboot的文件，静静的躺着。对于Android开发者来说，adb命令行肯定不会陌生.

3. 验证效果

下面的步骤，在Github中已经详细的列出。凑篇幅我再写一遍，顺便加一点自己的使用心得。

首先，你要下载一个hid-gadget-test文件。

模拟键盘的时候，如果终端上出现了a,恭喜你成功了一半。
模拟鼠标的时候，如果鼠标的指针移动了，恭喜你成功了。
嗯，如果你用的是Windows、macOS 系统，然后你就发现，为什么我的鼠标 没！有！动！静！

然后开始了我的爬坑之路。

爬坑之路

进行完上面的步骤，基本确定这是我们想要的功能，于是我去直接了修改公司系统代码的Kernel(注意，不是Nexus 5了，是公司在做的机器的系统源码)，编译，刷机。当时内心是抱着一丝侥幸，鼠标的问题，只在某些机型上存在呢！结果还是GG。然后从程序的角度分析，我做了以下尝试(注意，这个时候我已经解决了kernel 修改的问题，kernel 修改的步骤会后面跟大家分享)：

  1. 把对键盘的支持干掉，只保留了对鼠标的支持
  2. 看了一些HID协议相关的资料，修改了鼠标的hidg_func_descriptor
  3. 请教了一个做硬件的学长
  4. 等等
    这些步骤下来，然而问题并没有解决，尤其是修改HID，导致了一些error，明显是行不通的。怎么办呢？歇一歇，换换脑，去做其他的功能吧~于是我滚去开发写画的笔触效果，but 还没写两天，就被揪回来了。
    嗯 ，又是查了一天的资料。忽然看到一个USB Monitor 的东西，握草，USB 通信还能像HTTP请求被抓包。抓包的事，学长好像也提过一次。于是抱着试一试的心里，下载了USB Monitor，我先后抓取了真正的鼠标设备左键单击和Nexus 5模拟左键单击的数据，对比之后，豁然开朗。 前者(01 00 00 00 00)有5个byte,而后者只有3个byte(01 00 00)。也许是这个原因导致的。

我们真正和PC通信时发送的是byte为单位的数据, 上面的测试中，是用到了一个hid-gadget-test的文件做转换。
那就试一试直接写byte? 前面的爬坑之路的，虽然没有解决问题，但是却让我对这个项目足够熟悉。我知道hid-gadget-test操作的目标文件夹是/dev/hidg0 /dev/hidg1。
然后就是一波操作echo -e -n "\x0\x32\x0\x0 > /dev/hidg1",这个命令的作用是让鼠标指针移动50个像素。这里为什么是4位呢，经过测试，发现4位更合适。5位会触发PC读取两次数据，分别是4位和1位，这样会导致鼠标紊乱。\x 后的数字表示的是十六进制数据，第一个\x 后的数值表示鼠标，第二个\x后的数值表示x轴移动距离，第三个\x后的数值表示Y轴移动距离，最后一个是滑轮，默认是0就好了。
嗯，我的鼠标移动了🤣🤣🤣我兴奋的都要拆桌子了。没高兴太就，后面又遇一坑。
然后我又一波操作echo -e -n "\x0\x0xffffff80\x0\x0 > /dev/hidg1，这次换个负值试试！Boom!炸了，鼠标指针居然沿着Y轴移动了。我去查了echo这个命令的使用，\x这个后面只能设置2位16进制数字。。。怎么办呢，我打印了所有[-128,-1]这个区间数字16进制的表示，发现它们有一个共同的规律，它们的前六位都是ffffff,我猜测了一下传递负值应该怎么写，比如-128 = 0xffffff80, 取负数的后两位，echo -e -n "\x0\x80\x0\x0 > /dev/hidg1即可。最后试一试，果然成功。
到此为止，填平所有坑，可以愉快的编译内核，写代码了。

功能集成

Patch Kernel

内核打补丁步骤按照这个文件来：https://github.com/pelya/android-keyboard-gadget/blob/master/HTCM7_patch_steps
做这一步的时候，我先去完整的翻了一遍Repository的每一个文件夹。找到有用的东西，就记下来，找到最有用的就是这个文了，它有很详细修改kernel的步骤。按照步骤走，整个过程比较顺利，虽然有错误出现几次，但都不是需要花费以天为单位去解决的问题。
我需要的是3.10的patch，所以我把这个目录https://github.com/pelya/android-keyboard-gadget/tree/master/patches/existing_tested下所有3.10的patch包复制到一个文件夹。为什么要这样做呢？是因为不同机器的Kernel，即使版本相同，代码是有差别的。我原本的打算是，一个一个的尝试，最后在出错最少的那个包上做修改。可能是幸运加持，我使用的这个patch包htc_______flounder_____kernel_3.10.patch很顺利的完成了这个步骤。
修改完毕后，执行编译命令，就获得了kernel.img，大概6.6M。
